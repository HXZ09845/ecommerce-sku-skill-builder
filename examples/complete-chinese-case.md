# Complete Chinese Case: 从商品卖点到可执行视频镜头

This example is fictional and safe to publish. It uses a foldable desktop humidifier to show how `sku-skill-builder` changes a loose prompt workflow into a structured SKU Skill workflow.

这个案例回答三个问题：

1. 输入材料是什么。
2. Skill 中间具体补了什么判断。
3. 最终 prompt 为什么比原来更稳定。

## 0. 输入材料

假设用户已经用自己的业务 Skill 做完了初步判断，给到以下材料。

### 商品 Brief

| 项目 | 内容 |
|---|---|
| 商品 | 折叠桌面加湿器 |
| 场景 | 办公桌、熬夜加班、电脑旁 |
| 人群 | 长时间办公、桌面空间有限、想要安静补湿的人 |
| 视频目标 | 生成 9:16 真实感商品展示短视频 |
| 风险 | 商品形态漂移、雾气像烟、桌面比例失真、按钮/水箱细节变形 |

### 已判断好的卖点

| 卖点 | 业务判断 | 需要被画面证明的点 |
|---|---|---|
| 折叠小巧 | 适合桌面空间有限的人 | 放在电脑旁不占地方，折叠状态清楚 |
| 静音出雾 | 适合办公室和夜间使用 | 雾气轻、稳定，不打扰办公氛围 |
| 透明水箱 | 用户能看见水位 | 近景能看到水箱和水线 |
| 一键操作 | 新用户也能快速使用 | 手指靠近按钮，随后出现出雾结果 |

### 已有素材编排

| 素材 | 优先级 | 应控制什么 | 不应控制什么 |
|---|---:|---|---|
| `@Image1` 商品白底图 | P0 | 产品身份、外形、颜色、折叠形态 | 办公场景、人物动作 |
| `@Image2` 水箱细节图 | P1 | 透明水箱、水线、局部结构 | 整体比例、桌面环境 |
| `@Image3` 办公桌参考图 | P2 | 桌面、电脑、键盘、办公光线 | 产品身份、产品状态 |
| `@Video1` 办公桌推近参考 | P3 | 镜头节奏、轻微手持感 | 产品外观、品牌、背景布局 |

## 1. 普通 Prompt 的问题

用户可能原来会写：

```text
请生成一个高端真实的办公室桌面加湿器视频，展示产品折叠小巧、静音出雾、透明水箱和一键操作。参考商品图和办公桌视频，画面要真实、高级、适合电商带货，9:16。
```

这个 prompt 能跑，但不稳定，因为它把几个不同决策混在一起了。

| 问题 | 可能后果 |
|---|---|
| 多个卖点同时塞进一个镜头 | 模型平均用力，每个卖点都不够清楚 |
| 没有说明素材控制边界 | 办公视频可能污染产品外形，商品图也可能污染场景 |
| 没有锁定产品状态 | 折叠、打开、出雾、按键动作可能互相打架 |
| 没有说明结束条件 | 镜头容易继续漂移，产品越变越不对 |
| 没有失败回收方式 | 生成坏了以后只能继续改 prompt，经验不能复用 |

## 2. Skill 补上的中间判断

`sku-skill-builder` 不直接把输入材料拼成一段长 prompt。它先把生产判断拆开。

### 商品理解卡

| 字段 | 判断 |
|---|---|
| 主体 | 折叠桌面加湿器 |
| 关键部件 | 折叠机身、透明水箱、出雾口、按钮 |
| 稳定身份 | 外形、颜色、机身比例以 `@Image1` 为准 |
| 可变化状态 | 折叠静置、注水后、开机出雾、按钮触发 |
| 不应变化 | 品类不能变成香薰机、音箱、水杯或台灯 |

### 组件/状态矩阵

| 状态 | 可用镜头 | 禁止混用 |
|---|---|---|
| 折叠静置 | 桌面摆放、空间对比 | 不要同时出现按键动作和强出雾 |
| 开机出雾 | 雾气轻、稳定、从出雾口出现 | 不要让雾气从机身侧面或桌面出现 |
| 水箱近景 | 水线、透明材质、局部结构 | 不要生成可读品牌字或虚假刻度 |
| 一键操作 | 手指接近按钮，结果切到出雾 | 不强行生成复杂按压细节 |

### A/B 决策

| 镜头 | 类型 | 原因 |
|---|---|---|
| 桌面小巧 | A-out | 产品来自商品图，场景来自办公桌图，需要外部场景融合 |
| 静音出雾 | A | 主要是产品自身状态变化，减少场景干扰 |
| 透明水箱 | A-in | 依赖产品局部细节，尽量少换环境 |
| 一键操作 | A | 动作风险较高，用“意图动作 + 结果状态”更稳 |

## 3. 前后 Prompt 对比

| 原始写法 | Skill 改写后 | 为什么更好 |
|---|---|---|
| 生成一个高端真实视频 | 真实感竖屏手机拍摄，办公桌自然光，轻微手持推近 | 风格变成可执行摄影语言 |
| 展示折叠小巧、静音出雾、透明水箱、一键操作 | 每个 Unit 只证明一个卖点 | 避免卖点互相抢画面 |
| 参考商品图和办公桌视频 | `@Image1` 控制产品身份，`@Image3` 控制办公场景，`@Video1` 只控制镜头节奏 | 避免素材互相污染 |
| 画面高级 | 产品稳定、可检查、桌面比例正确、无多余文字 | 把审美词变成验收标准 |
| 失败后继续改 prompt | 记录失败原因，回收到素材角色、状态限制或 validator | 让经验进入工程机制 |

## 4. 输出的 Prompt Plan

下面是两个示例 Unit。真实项目里会继续扩展到更多镜头，并配套 validator。

### Unit U01: 折叠小巧办公桌

| 字段 | 内容 |
|---|---|
| 卖点 | 折叠小巧 |
| 画面目标 | 产品放在电脑和键盘旁，能看出不占空间 |
| 类型 | A-out |
| 时长 | 5s |
| 核心素材 | `@Image1` + `@Image3` + optional `@Video1` |
| 人要确认 | 桌面参考是否能代表目标人群；产品大小是否允许被放大展示 |

```text
【素材说明】
@Image1：P0，控制加湿器产品身份、折叠机身形态、颜色和主要比例；不控制办公桌背景。
@Image3：P2，控制办公桌、笔记本电脑、键盘、桌面光线；不控制产品身份和产品状态。
@Video1：P3，仅控制轻微手持推近的镜头节奏；不控制产品外观、品牌、场景布局。

【画面要求】
生成 9:16 真实感手机拍摄的办公桌商品视频。加湿器必须保持 @Image1 的折叠形态、颜色和机身比例，被放在 @Image3 风格的电脑和键盘旁边。镜头从桌面中近景轻微手持推近，最终停在产品稳定、足够大、能看清折叠机身的位置。这个镜头只证明“折叠小巧，不占桌面空间”，不要同时表现按钮操作和大范围出雾。不要生成可读品牌字、字幕、水印或额外 UI。

【验收点】
产品身份不变；折叠状态清楚；桌面比例自然；产品没有被电脑或键盘遮挡；镜头结束时产品可检查。
```

### Unit U02: 静音轻雾

| 字段 | 内容 |
|---|---|
| 卖点 | 静音出雾 |
| 画面目标 | 轻雾稳定出现，办公氛围不被打扰 |
| 类型 | A |
| 时长 | 5s |
| 核心素材 | `@Image1` |
| 人要确认 | 是否允许用画面暗示“静音”，还是需要旁白/字幕承接 |

```text
【素材说明】
@Image1：P0，控制加湿器产品身份、机身颜色、折叠后轮廓、出雾口位置；不控制背景陈设。

【画面要求】
生成 9:16 真实感桌面近景视频。加湿器保持 @Image1 的产品外形和折叠状态，放在安静办公桌环境中。镜头固定或极轻微推近，出雾口出现短而细的稳定轻雾，雾气只从正确出雾口出现，不要变成浓烟、蒸汽爆发或扩散到整个画面。画面氛围安静克制，电脑屏幕和键盘只作为背景信息，不抢主体。不要生成可读文字、字幕、水印或虚假数值。

【验收点】
雾气轻、短、稳定；出雾位置正确；产品轮廓不变；办公氛围不被戏剧化。
```

## 5. 人在流程中参与什么

这个 Skill 不是把人排除在外，而是把人的经验放到关键闸口。

| 人的参与点 | 为什么必须由人判断 |
|---|---|
| 确认卖点优先级 | 哪个卖点更重要通常来自业务目标，不是模型自己知道 |
| 确认素材角色 | 一张图是控制产品、场景、动作还是质感，需要运营经验 |
| 确认 A/B 类型 | 是否换景、是否用动作视频参考，会影响生成稳定性 |
| 确认每个卖点的 visual proof | 画面有没有真的证明卖点，需要人的商业判断 |
| 审查失败结果 | 失败是身份漂移、状态冲突、素材污染还是动作过难，需要归因 |
| 批准规则沉淀 | 哪些失败要写入 gotchas、validator、trigger tests，需要负责人定稿 |

## 6. 失败修复如何工程化

| 失败现象 | 根因判断 | 修复方向 | 沉淀位置 |
|---|---|---|---|
| 产品变成香薰机 | 品类身份不够锁定 | 强化 P0 身份控制和禁止品类漂移 | `gotchas.md` / validator |
| 雾气像烟 | 动作描述过强 | 改成短、细、轻、稳定，只从出雾口出现 | prompt-plan rule |
| 桌面比例不自然 | 场景图控制过强 | 降低场景素材权重，增加产品可检查结束条件 | asset manifest |
| 按钮变形 | 动作细节过难 | 用“手指接近按钮 + 结果状态”代替强按压 | product rule |
| 多卖点都不明显 | 一个镜头目标过多 | 每个 Unit 只证明一个卖点 | workflow rule |

## 7. 最终差别

| 维度 | 原来 | 使用 Skill 后 |
|---|---|---|
| 工作对象 | 一段 prompt | 可验证的 SKU Skill 包 |
| 素材关系 | “参考一下” | 明确 `controls` / `does_not_control` |
| 卖点表达 | 口号式描述 | 每个卖点绑定 visual proof |
| 视频生成 | 多段靠手感续写 | Unit 化、多段生成、续写和修复 |
| 失败处理 | 临时改词 | 回收到规则、validator、gotchas |
| 可交接性 | 依赖个人经验 | 文件化、结构化、可复用 |

这个项目真正解决的不是“帮你写一句更漂亮的 prompt”，而是把你已经判断好的商品卖点和素材编排，翻译成模型更容易执行、团队更容易复用、失败更容易修复的视频生产机制。
