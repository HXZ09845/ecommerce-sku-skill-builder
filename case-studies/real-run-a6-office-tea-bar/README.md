# Real Run Case Study: Office Overtime Tea-Bar Machine

This is an anonymized real production run. The original project used a real ecommerce product, real reference assets, and a real prompt-plan workflow. To keep the public repository safe, product brand names, raw media files, asset IDs, local paths, and unpublished generated videos are not included.

What is preserved here is the reusable engineering evidence:

- Unit schedule and scene logic.
- A/B generation decisions.
- Reference asset role boundaries.
- Before/after prompt changes.
- Take-review and failure repair rules.
- Rules that should be written back into future SKU Skills.

## 2. Original Unit Schedule

| Unit | Type | Duration | Purpose | Selling point / role |
|---|---:|---:|---|---|
| U1 | A | 10s | Establish office overtime pain and introduce product | Scene opening |

| U3 | A | 10s | Show remote control from the desk | Remote operation |

## 3. Why The Original Prompt-Plan Was Already Valuable

The original workflow had already done a lot of expert work:

| Existing strength | Why it matters |
|---|---|
| Scene-specific micro-actions | The office overtime scene did not feel generic |
| Dedicated props | Desk, notes, cup, keyboard, delivery box, and work ID created scene proof |
| A/B Unit schedule | Action-video units and image-to-video units were separated |
| Environment anchor | Product position and office layout had one stable spatial source |
| Selling-point mapping | Each Unit knew which business claim it was responsible for |
| Self-review gates | Narrative quality was checked against explicit criteria |

The SKU Skill Builder does not erase this work. It compiles it into stricter shot contracts.

## 4. What The Builder Added

| Layer | Original prompt-plan | Builder output |
|---|---|---|
| Asset roles | Materials were listed with useful notes | Each ref gets `controls only` and `must not transfer` boundaries |
| Clip scope | Timeline described many actions | Each Unit gets `This clip only` and a single proof target |
| Endpoint | Some clips ended by narrative continuation | Each clip gets a visible `Stop when` condition |
| Continuity | Product position was described globally | Continuity locks become project-state rules |
| Risk control | Some high-risk actions remained in timeline | Risky actions are converted to intent cue plus result state |
| Failure review | Failures could be fixed manually | Take-review table assigns first repair variable per Unit |

## 5. Project-State Capsule

```text
PROJECT ID: office_tea_bar_overtime_case

STORY GOAL:
The user is working late, wants water or a hot drink without breaking focus,
uses the tea-bar appliance to reduce movement and waiting, then returns to work.

FINAL OUTCOME:
The user sits back at the desk with a hot drink visible. The appliance remains
stable in the office background, supporting the claim that office hydration no
longer interrupts work.

CONTINUITY LOCKS:
- Product stays in the same right-side wall position in the office.
- Office desk, partition, chair, monitor, and neutral lighting remain consistent.
- Exact product text, screen text, brand labels, and numeric claims are not
  generated by the model; they should be handled by VO, subtitles, or post.

RESERVED BEATS:
- U1 cannot show auto-fill, remote control, sealed structure, or fast heating.
- U2 only proves auto-fill and water-full stop.
- U3 only proves remote intent and machine response, not the whole boil process.
- U-bridge is lifestyle warmth, not a product-function proof.
- U4 proves clean water and material trust visually; exact sealed-water claims
  belong to VO or subtitle.
- U5 proves hot drink ready and work resumed; exact wattage belongs to VO or subtitle.
```

## 6. Take-Review Table

| Unit | Accept criteria | Common failure | First repair variable |
|---|---|---|---|
| U1 | Office pain is clear; product position is stable; no later selling point appears early | Character performs too many cup/water actions or product drifts | Narrow to look at empty cup -> turn to product -> arrive beside product |
| U2 | Action video timing is preserved; background is office bokeh only | Environment anchor overwrites video composition or adds people/buttons | Make environment asset background-tone only; strengthen strict video replication |
| U3 | User presses remote from desk; product remains stable in distance | Prompt tries to show the entire boiling process | Keep only remote intent and product response |
| U-bridge | Warm lifestyle beat; kettle shape stable; result state clear | Long continuous pour creates hand/water/kettle distortion | Use intent cue -> result state |
| U4 | Clean water/material trust appears; whole product identity remains stable | Detail images overwrite whole-product identity or create fake section diagrams | Limit detail refs to material texture only |
| U5 | User returns to work; hot drink visible; product fixed in background | Model generates numeric wattage, fake link text, or brand text | Move precise claims to VO/subtitle/post |

## 7. Rules Written Back From This Run

| Rule candidate | Where it belongs |
|---|---|
| Every Unit needs `one visible beat` and `Stop when` | `prompt-plan-format.md` |
| B-class action-video prompts must preserve source motion and timeline | `product-rules.md` / `gotchas.md` |
| Long hand actions should use intent cue plus result state when no reliable action video exists | product-specific `gotchas.md` |
| Exact wattage, brand labels, link text, and screen text should not be model-generated | `product-rules.md` |
| Multi-clip scripts need reserved beats so early clips do not leak later selling points | `audio-visual-sync.md` |
| Generated clips should be reviewed before prompt changes; first repair variable must be identified | `trigger-tests.md` |
